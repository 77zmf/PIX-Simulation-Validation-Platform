# AI Superbody Execution Closure Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a repeatable stable validation workflow that turns PIX simulation runs into final KPI results, reports, cleanup evidence, and developer-ready bugpacks.

**Architecture:** Reuse the existing `simctl` control plane and stable stack instead of creating one-off scripts. The implementation starts with host recovery and BEV run closure, then runs a small planning/control campaign, generates bugpack evidence for failures, and updates digest/owner handoff artifacts. NoMachine is not part of the execution path; host access is SSH/Tailscale only.

**Tech Stack:** Python 3 `simctl`, ROS 2 Humble, Autoware Universe, CARLA 0.9.15, SUMO, Bash, YAML scenarios, KPI gates, report/bugpack artifacts.

## Execution Rules

- Use SSH/Tailscale only. Do not use NoMachine as an automation dependency.
- Do not start a new run if `/data/pix` has less than 80 GiB free, available memory is below 2 GiB, swap free is below 8 GiB, or ports `2000`, `8000`, or `9000` are occupied by another user.
- Treat `launch_submitted` as incomplete.
- A target is closed only when `run_result.json` has final `passed` or `failed`, KPI output exists, report/replay paths exist, and cleanup leaves ports/processes clear.
- If a run fails due planning/control behavior, produce a bugpack and do not patch Autoware behavior.
- If a failure is simulator, vehicle model, perception, host, or asset related, fix the infrastructure issue only when the root cause is deterministic and repo/local-host scoped.
