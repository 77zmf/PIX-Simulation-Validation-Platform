# AI 超体计划补充：感知专题工作与主线计划并行推进说明

本文用于补充《AI超体计划-自动驾驶仿真验证平台建设》中的感知专题工作描述，目的是把超体计划主线与感知专题工作流分开表述、并行推进、统一验收。本文不替代原项目计划书，而是作为项目级补充说明，便于汇报、立项沟通和阶段对齐。

## 一、项目定位

`zmf_ws` 是 AI 超体计划的主工作区，承担稳定闭环验证平台建设的主线任务，重点覆盖：

- `simctl` 控制平面与执行链路统一
- Ubuntu 22.04 主机环境准备与运行基线
- Autoware Universe + ROS 2 Humble + CARLA 0.9.15 稳定闭环
- 公开道路资产束、场景模板、KPI gate、digest 和报告闭环

`zmf_bev` 是感知专题工作区，承担 BEVFusion 感知基线、数据校准、距离提升和 shadow 感知评测准备等专题能力建设任务，重点覆盖：

- BEVFusion + nuScenes 感知评测基线
- 公开道路和车端数据的校准检查与数据健康验收
- 远距离感知问题归因与距离提升方案
- 为 UniAD-style shadow / VADv2 对照实验提供感知输入、评测口径和专题结论

边界说明如下：

> 超体计划负责项目级组织与交付闭环，感知工作负责专题能力建设与评测基线，两者并行推进、相互支撑。

## 二、阶段目标中的感知工作落位

感知工作不作为超体计划的唯一主线，而是作为专题工作包并行挂载到 Phase 2 和 Phase 3 的关键交付中。

| 阶段 | 时间目标 | 超体计划主线目标 | 感知专题工作目标 |
| --- | --- | --- | --- |
| Phase 1 MVP / 跑通核心流程 | 2026-04-05 | 完成 Ubuntu 主机准备、`simctl run -> run_result.json -> report -> replay` 最小闭环 | 完成感知链路现状梳理，明确 `mmdetection3d_pix` 与 `BEVFusion-ROS-TensorRT` 的训练-导出-部署关系，为专题排查建立基线 |
| Phase 2 功能完善 / 稳定性提升 | 2026-05-03 | 打通稳定闭环、批量回归、KPI 门禁和报告流程 | 完成数据校准规范、车端与 nuScenes 数据健康检查、远距掉检问题归因、短期优化闭环 |
| Phase 3 优化上线 / 收尾交付 | 2026-06-14 | 完成公开道路资产沉淀和统一验证流程 | 完成 BEVFusion 感知评测基线、距离提升中长期路线、为 shadow 感知输入和评测提供稳定专题交付 |

## 三、感知子方向定位

感知专题工作按“数据可信、链路可信、评测可信、能力可提升”四层推进。

### 1. 数据可信

目标是确保 nuScenes、公开道路资产和车端实测数据在时间同步、内外参、坐标系、相机顺序、点云字段和标注投影上可追溯、可解释、可复查。

### 2. 链路可信

目标是打通并固化以下链路：

- 训练链路：`mmdetection3d_pix`
- 导出链路：checkpoint -> ONNX -> TensorRT engine
- 部署链路：`BEVFusion-ROS-TensorRT`
- 评测链路：fillback / 离线可视化 / 车端话题输出 / Autoware 感知消息

### 3. 评测可信

目标是避免只看总体 mAP/NDS，而忽略真实业务关注的远距离效果。专题评测将补充以下口径：

- 距离分桶召回与置信度分布
- 阈值前候选框数与阈值后保留框数
- 图像投影与 BEV 对齐证据
- 车端静态标定检查与直道远距目标案例

### 4. 能力可提升

目标不是仅说明“当前 108m 配置无效”，而是形成：

- 短期可落地的排查与优化动作
- 中长期可建设的远距离感知路线
- 可被项目管理吸纳的阶段里程碑和验收项

## 四、关键交付补充

在原项目计划书交付项基础上，新增以下感知专题交付。

### 1. 数据校准规范

形成《感知工作说明：BEVFusion 数据校准与感知距离提升》中的第一章内容，明确：

- 为什么需要对 nuScenes、车端和公开道路数据做校准检查
- 必须重点检查的 8 类数据对象
- 当前仓库链路中的 4 个特殊风险点
- 数据校准验收表模板和证据要求

### 2. BEVFusion 感知评测基线

形成一套专题级基线认知：

- 当前训练仓与部署仓的参数关系
- `108m` 配置在训练侧和部署侧分别代表什么
- nuScenes 标准监督半径与真实业务目标之间的差距
- 哪些指标可作为专题基线，哪些不能直接代表车端远距能力

### 3. 距离提升方案

围绕“训练参数显示 108m，但车上实测约 70m”形成两层方案：

- 短期方案：优先排查现有链路损失，确认问题发生在监督、标定、导出、部署还是后处理阶段
- 中长期方案：构建真正面向远距离业务目标的数据、指标和训练策略

### 4. Shadow 感知评测准备

为后续 UniAD-style shadow / VADv2 对照实验提供前置条件：

- 感知输入定义
- 评测距离口径
- 典型场景与 corner case 证据沉淀
- 与稳定主线不冲突的专题推进边界

## 五、主要风险补充

在原计划书主要风险基础上，建议增加以下感知专题风险描述。

| 风险 | 等级 | 表现方式 | 应对思路 |
| --- | --- | --- | --- |
| 数据质量风险 | 高 | 时间同步、文件完整性、标注投影或点云字段异常会直接导致感知基线失真 | 先做数据健康检查与投影验收，不在数据不可信的前提下继续做模型结论 |
| 坐标链路风险 | 高 | nuScenes、车端、可视化和部署链路中存在多处坐标转换，容易重复修正或修正不一致 | 固定单一坐标口径，明确“哪一层负责变换”，逐层去重 |
| 真实域差异风险 | 高 | nuScenes 标准训练结果无法直接代表车端真实摄像头、点云密度和场景分布 | 采用 fillback + 车端实测对照，区分训练域问题和部署域问题 |
| 远距能力不足风险 | 高 | 输入覆盖范围扩到 108m 不等于有效监督和有效检测都能覆盖 108m | 增加距离分桶评测，先找掉检主因，再推进远距能力建设 |

## 六、项目级验收口径与感知专题验收口径

超体计划主线验收与感知专题验收要区分层级，避免错把专题问题当作整个平台是否成立的唯一标准。

### 1. 项目级验收

项目级验收继续围绕平台能力展开：

- `simctl` 命令链可执行
- 稳定闭环可重复运行
- `run_result.json -> KPI gate -> report -> replay` 可形成闭环
- 公开道路资产和场景能够进入统一验证流程

### 2. 感知专题级验收

感知专题验收建议采用以下四条：

- 数据校准通过：时间、坐标、投影、相机顺序、点云字段检查完成并留证
- 远距问题完成归因：明确 70m 附近掉检的主因层级，而不是只有现象描述
- 短期方案形成可执行闭环：可按步骤复现排查、输出对比结果
- 中长期方案形成里程碑任务：明确数据、训练、指标和部署升级方向

## 七、建议纳入原计划书的位置

建议把本文内容吸收到原计划书的以下位置：

- 在“二、三阶段交付计划”中，把感知专题工作作为 Phase 2/3 的并行工作包
- 在“3.2 技术选型”中，补充 `zmf_bev / BEVFusion / TensorRT / ROS2` 的专题定位
- 在“3.3 关键技术难点 & 应对思路”中，加入数据校准和远距能力不足的专题风险
- 在“六、主要风险”中，增加数据质量、坐标链路、真实域差异和远距能力四类风险

## 八、补充结论

当前建议采用“双轨推进”方式：

- `zmf_ws` 继续承担平台主线，优先保证闭环、自动化、资产标准化和 KPI gate
- `zmf_bev` 作为感知专题线，优先完成数据校准规范、BEVFusion 评测基线和距离提升路线

这样可以避免两类常见问题：

- 把感知问题泛化为整个平台不可交付
- 把平台主线问题误写成纯感知模型问题

双轨推进的目标不是分裂交付，而是通过明确边界、专题验收和统一口径，让平台主线与感知专题相互支撑、互不拖拽。
