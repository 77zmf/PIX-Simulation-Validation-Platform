# 项目推进团队与边界

当前项目对外展示名统一为 `PIX Simulation Validation Platform`。  
当前可用 GitHub 仓库路径仍然是 `pixmoving-moveit/zmf_ws`。

## 1. 团队目标

这支团队的目标不是单点完成若干脚本，而是持续推进一条可复用的仿真验证主线：

- `Autoware Universe + CARLA 0.9.15 + UE4.26`
- 自动化验证与报告
- 公开道路资产与 corner case 沉淀
- E2E shadow 研究，但不破坏主线交付

## 2. 角色与职责

### 朱民峰

- 负责：稳定主线、Ubuntu 主机、控制平面、自动化、KPI、季度交付节奏
- 输出：稳定栈运行结果、闭环验收、自动化链路、周推进判断

### 罗顺雄 / lsx

- 负责：公开道路地图点云资产、重建输入、corner case、现场问题复现输入
- 输出：资产束、场景模板、问题案例索引、场景成功信号

### 杨志朋 / Zhipeng Yang

- 负责：`BEVFusion` 感知基线、`UniAD-style / VADv2` shadow 研究、规划/感知接口比较
- 输出：研究 profile、研究场景、shadow 指标和对照结论

### Codex PMO 支持位

- 负责：digest、周会材料、阻塞项聚合、GitHub/仓库侧管理同步
- 输出：每日提醒、周会摘要、风险清单、任务口径统一

## 3. 明确边界

### 朱民峰的边界

- 负责拍板：稳定主线、四月初硬门槛、主链路验收
- 不替代：资产整理、研究线细节实现、单个场景的数据制作

### 罗顺雄 / lsx 的边界

- 负责拍板：资产束结构、场景输入完整性、corner case 清单
- 不替代：稳定栈联调、控制平面实现、算法路线拍板

### 杨志朋 / Zhipeng Yang 的边界

- 负责拍板：`BEVFusion` 基线接入方式、shadow 指标草案、研究对照设计
- 不替代：稳定主线环境搭建、资产标准化、季度主验收

### Codex PMO 支持位的边界

- 负责推动：提醒、节奏、任务聚合、材料整理
- 不负责拍板：技术路线、真实环境调试、最终验收

## 4. 交接规则

- 所有执行任务必须能落到明确 owner
- 所有阻塞项必须写入 GitHub Project 或 digest
- 所有研究结论必须能回指到具体场景和 KPI
- 所有周推进动作必须能回答“本周要交付什么、还卡在哪里、谁来解”
