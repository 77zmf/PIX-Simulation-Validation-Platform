# 项目推进团队与作战机制

## 1. 团队定位

这个团队不是新增一个组织架构，而是把现有项目成员和 Codex 支持能力编组成一个稳定推进单元，目标是：

- 把稳定主线尽快推进到真实可运行状态
- 把任务、场景、阻塞项和交付物收敛到统一节奏
- 把周推进从“人盯人”变成“板子 + digest + 周会 + 自动提醒”共同驱动

## 3. 明确边界

### 朱民峰的边界

- 有权决定：稳定主线优先级、主链路验收口径、阶段性交付是否过关
- 必须亲自抓：公司 Ubuntu 主机环境、Autoware + CARLA 主链路、KPI 和四月初硬门槛
- 不替代：公开道路资产整理、远端 UE5 / E2E 预备的具体执行
- 必须升级的情况：P0 延期、主机权限受限、真实闭环无法推进

### 罗顺雄 / lsx 的边界

- 有权决定：公开道路资产束结构、corner case 录入方式、现场问题到场景模板的转换方案
- 必须亲自抓：地图点云资产、重建输入、首批公开道路场景模板
- 不替代：稳定栈联调、感知/E2E 算法路线拍板
- 必须升级的情况：资产缺失、字段不全、场景无法进入统一回归链路

### 杨志朋 / Zhipeng Yang 的边界

- 有权决定：`BEVFusion` 基线接入方式、shadow 指标草案、UE5 远端预备方案
- 必须亲自抓：感知基线与 shadow 输入映射、`UniAD-style` / `VADv2` 评测口径、远端实验前置条件
- 不替代：稳定主线环境联调、公开道路资产标准化拍板
- 必须升级的情况：远端 GPU 资源不明确、shadow 输入契约不成立、对照路线无法比较

### Codex PMO 支持位的边界

- 有权推动：任务梳理、文档统一、digest 汇总、周会材料、阻塞项显式化
- 必须做到：每天把 overdue、blocked、due soon 和 owner 级 next actions 收敛出来
- 不替代：技术负责人拍板、真实环境调试、最终交付验收
- 必须升级的情况：板子与实际不一致、owner 不明确、风险项没有下一动作

## 4. 交接规则

- 任何任务只允许 1 个主 owner，其他人只能是协作，不允许多人共同模糊负责
- 任何阻塞项必须带 owner、当前状态、下一动作和预计解除时间
- 任何跨边界事项先在周会或 digest 中显式化，再决定是否切换 owner
- Codex PMO 只负责推动和提醒，不默认改写 owner 责任

## 5. 推进机制

### 每日推进

- 工作日早上检查任务看板、场景看板、最新 digest、最新验证结果
- 输出 overdue、due soon、blocked 和 owner 级 next actions
- 如果四月初硬门槛有风险，必须显式标红

### 每周推进

- 周一：启动周目标，确认 P0 清单和本周验收口径
- 周三：专门清理阻塞项，检查跨人协作和外部依赖
- 周五：收敛本周交付、验证结果和下周准备项

### 风险升级

- 任一 P0 任务延迟超过 1 次周会，立即升级
- 任一阻塞项没有 owner 或下一动作，视为未管理
- 新出现的现场问题必须先进入 Scenario Backlog，再决定是否进入主线

## 6. 团队当前硬目标

短期硬门槛：

- `2026-04-05` 前完成公司 Ubuntu 主机环境搭建
- 打通 `simctl run -> run_result.json -> report -> replay` 数据闭环

在这个门槛完成前，团队只优先做 5 件事：

1. 公司 Ubuntu 主机访问、权限和基础依赖准备
2. `CARLA 0.9.15` 启动验证
3. `Autoware Universe` 工作区准备与首次编译
4. `autoware_carla_interface` 最小链路打通
5. 第一条 L0 smoke 路线产出真实 `run_result.json`

## 7. 团队工作入口

- GitHub Task Board：公开任务执行镜像
- GitHub Scenario Board：公开场景执行镜像
- Digest Inbox：自动提醒入口
- Notion Program Board：详细任务和 owner 管理
- Notion Scenario Backlog：场景清单和优先级
- Weekly Review Dashboard：周会和节奏管理

## 8. Codex 如何持续推进

Codex PMO 支持位会重点做这些事：

- 每日聚合任务、场景、验证状态和阻塞项
- 每周生成周会前置材料和 owner 级 next actions
- 当你要求同步时，更新仓库文档、GitHub Project 和 Notion 页面
- 把散乱的信息重新收敛成项目可执行口径

这意味着：

- 人负责拍板、联调、交付
- Codex 负责汇总、提醒、推进和管理负担减轻

## 9. 判断这支团队是否有效的标准

- 任务 owner 清楚，不存在“大家都知道但没人负责”
- P0 延期和阻塞项都能在 1 个工作日内被识别
- 每周都有新增交付物、报告或场景资产沉淀
- 项目书、看板、仓库文档和自动 digest 保持同一口径
